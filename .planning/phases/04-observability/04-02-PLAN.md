## Phase Goal

**As a** recruiter, **I want** completed runs to be saved and browsable after the fact, **so that** I can open any past run and inspect its full step trace, timing, and provider fallbacks — evidence the agent is observable, not a black box.

**This plan's slice:** the durable write path and the two read endpoints. A completed run is persisted to Supabase without blocking the answer (OBS-01); `GET /runs` lists a session's runs (OBS-02); `GET /trace/{run_id}` serves the full persisted trace with per-step timing (OBS-03); stored content is redaction-scrubbed and never leaves for an external SaaS (OBS-06). Consumes the richer trace data (per-step `elapsed_ms`, `usage.fallback_events`) that Plan 04-01 produces.

## Acceptance Criteria

- [ ] A streamed run persists a `traces` row *after* the final SSE event; persist failure is swallowed and never affects the answer (OBS-01).
- [ ] `GET /runs` returns the session's runs (thread_id scoped, `created_at DESC`), empty list when `pool is None` (OBS-02).
- [ ] `GET /trace/{run_id}` reads DB-first with in-memory fallback and 404 when neither exists; persisted steps carry `elapsed_ms` (OBS-03).
- [ ] Persisted `steps`/`final_answer` are redaction-scrubbed; no external trace SaaS is enabled (OBS-06).
- [ ] Both read routes are double-registered, `@limiter.exempt`, and `/runs` has a `vercel.json` rewrite; full suite green.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| X-Session-Id header / run_id path param → SQL | user-controlled input reaches parameterized queries |
| stored trace content → Supabase | run steps/final_answer may embed a secret and must be scrubbed before persistence |
| run_id → cross-session trace read | a guessable/leaked run_id could expose another session's trace |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-4-03 | Tampering | traces SQL (persist + fetch) | high | mitigate | psycopg `%s` parameters only; never f-string run_id/thread_id/query into SQL (PersistTests asserts parameterized SQL) |
| T-4-04 | Information Disclosure | persisted steps/final_answer | high | mitigate | `redact_secrets` applied to stored step strings + final_answer before INSERT (RedactionTests) — OBS-06 |
| T-4-05 | Information Disclosure / Broken Access Control | `GET /trace/{run_id}` | medium | accept (documented) | run_id is a 32-hex UUID (not enumerable); the frontend only surfaces a session's own run_ids via session-scoped `/runs`. Cross-session read requires guessing a UUID. Accepted for a no-auth portfolio demo; a follow-up (V2) can scope `/trace` by thread_id. Flagged in 04-RESEARCH.md Open Question / Security Domain. |
| T-4-06 | Information Disclosure | `GET /runs` | high | mitigate | `WHERE thread_id = %s` from validated `_get_session_id`; no cross-session listing (RunsEndpointTests.test_runs_scoped_by_session) |
| T-4-07 | Denial of Service | read endpoints vs 10/min limit | low | mitigate | `@limiter.exempt` on both read routes (reads are cheap; a turn already spends its limited call on /run) |
| T-4-08 | Denial of Service | unbounded traces growth | low | accept | `fetch_runs` LIMIT bounds reads; table-level retention prune deferred (04-RESEARCH Open Question 3) |
