<objective>
Complete the conversation-persistence slice on the client: generate and persist an anonymous session id in localStorage, send it as the `X-Session-Id` header on every run, expose it from the `useAgent` hook, and surface it as a copyable chip in the chat status strip.

Purpose: 02-01 made the backend extract `X-Session-Id` into a `thread_id` and persist turns via the checkpointer. This plan supplies that header from the browser — turning the round-trip on — and makes the session id visible so a recruiter can verify cross-session persistence manually (SC4).

Output: A browser that maintains a stable session, restores conversation after reload (SC1), and shows the session id (SC4).
</objective>

## Phase Goal

**As a** returning user, **I want to** reopen the site and find my conversation still there (and see the session id that ties it together), **so that** I can pick up where I left off and verify the agent really remembers across sessions.

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-06 | Spoofing | session id generation | low | accept | `crypto.randomUUID()` yields a UUIDv4 (~122 bits entropy); non-sequential, not brute-forceable for a portfolio demo |
| T-02-01 | Tampering | header propagation | high | mitigate | Client always sends a well-formed UUID; the backend (02-01 `_is_valid_session_id`) still validates and rejects malformed values, so a tampered header cannot reach the namespace/SQL paths |
| T-02-ID | Information Disclosure | visible session id | low | accept | The id is meant to be copyable (SC4). Anyone who learns it can read that session's memory — acceptable for an anonymous demo; documented as the bearer-token tradeoff |
</threat_model>
