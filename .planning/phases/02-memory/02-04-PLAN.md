<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> API | `session_id` arrives as a URL path param on DELETE; attacker-controllable |
| API -> DB | clear deletes checkpoint rows (adelete_thread) and store rows (parameterized LIKE) for one namespace prefix |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Tampering | DELETE session_id path param -> raw store SQL | high | mitigate | Validate against the canonical UUID pattern (`_is_valid_session_id`); reject malformed with 400 before any DB access; the store delete uses a parameterized LIKE (value bound, not concatenated) so a UUID cannot inject wildcards |
| T-02-02 | Elevation of Privilege | cross-session deletion/access | high | mitigate | The delete is scoped to exactly one namespace prefix `memories.{session_id}` and one thread; UUIDv4 ids are unguessable, so no enumeration to a foreign session |
| T-02-DoS | DoS | unauthenticated DELETE spam | low | accept | Endpoint stays under the default 10/min per-IP slowapi limit; clearing one's own session is idempotent and cheap |
| T-02-SC | Tampering | dependencies | low | accept | No new packages introduced |
</threat_model>
